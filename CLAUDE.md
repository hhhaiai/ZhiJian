# 需求质检台 - 产品需求文档

## 一、产品定位

### 1.1 产品定义

**需求质检台** 是一款帮助产品经理提升需求文档质量的辅助工具。

### 1.2 核心价值

> **让产品经理在提交需求之前，先经过 AI 的"辅助检查"，把能发现的问题先标出来，减少开发过程中的反复确认。**

### 1.3 产品口号

```
写需求 → AI 辅助检查 → 补充完善 → 高质量交付
```

---

## 二、背景与痛点

### 2.1 当前问题

| 痛点 | 现状 | 影响 |
|------|------|------|
| 需求质量差 | 产品给的需求不完善，缺少审核环节 | 开发做到一半发现缺东西 |
| 反复确认 | 开发中、测试阶段还在确认需求 | 工期延误，研发情绪低落 |
| 需求说不清 | 产品常说"我想下"，以为不用说 | 返工、交付质量差 |
| 文档散落 | 需求在 Figma/Axure + 禅道，分散各处 | 找不到历史记录 |

### 2.2 期望改变

- 产品写完需求后，**先自己检查一遍**再提交
- AI 帮助标记**明显遗漏**的问题，但不是替代人工审核
- 需求文档**集中管理**，不会丢失

## 四、功能需求

### 4.1 MVP 功能（第一阶段）

#### 功能 1：需求录入

**目的**：统一需求格式，强制产品思考清楚

**录入字段**：

| 字段 | 必填 | 说明 |
|------|------|------|
| 需求标题 | 是 | 简单描述这个需求做什么 |
| 需求描述 | 是 | Markdown 格式，详细说明功能、流程、边界 |
| 设计图链接 | 否 | Figma/Axure 链接 |
| 优先级 | 是 | P0（紧急）/P1（高）/P2（中）/P3（低） |
| 需求类型 | 是 | 新功能 / 缺陷修复 / 体验优化 |

**描述模板（引导）**：

```
## 功能描述
[一句话说明这个功能做什么]

## 用户场景
- 场景1：用户 XXX 时，XXX
- 场景2：XXX

## 交互流程
1. 用户进入 XX 页面
2. 点击 XX 按钮
3. 系统 XX

## 边界/异常情况
- 网络异常如何处理
- 数据为空如何显示
- 权限不足如何提示

## UI 要求
[控件状态、动画效果等]
```

#### 功能 2：AI 辅助检查（核心）

**定位**：AI 是"辅助检查工具"，只提建议，不自动驳回

**检查逻辑**：

```
用户点击"AI 检查"
       ↓
调用 AI 分析需求描述
       ↓
返回"建议补充项"列表
       ↓
用户自行判断是否修改
       ↓
确认无误后，标记为"待开发"
```

**AI 检查项**：

| 检查项 | 说明 | 输出 |
|--------|------|------|
| 完整性 | 必填项是否填写 | "缺少边界情况说明" |
| 清晰度 | 描述是否说清楚 | "第2步的交互没有说清楚" |
| 逻辑闭环 | 流程是否完整 | "没有说明异常如何处理" |
| 分工明确 | 前后端各自要做什么 | "建议说明后端接口要返回什么" |

**输出示例**：

```
✅ 已填写完整
⚠️ 建议补充：
   1. 缺少网络异常时的提示文案
   2. 没有说明数据为空时的展示效果
   3. 建议补充后端接口返回格式说明
```

#### 功能 3：需求列表

- 展示所有需求（标题、状态、优先级、时间）
- 支持筛选（状态、优先级）
- 支持搜索（标题关键词）
- 按时间倒序排列

#### 功能 4：需求详情

- 完整展示需求信息
- Markdown 渲染
- AI 检查历史（每次检查的结果）
- 状态变更记录

### 4.2 状态流转

```
草稿 → 待检查 → （AI检查）→ 确认完善 → 待开发 → 开发中 → 已完成
              ↓
         标记"已检查，待完善"
```

| 状态 | 说明 |
|------|------|
| 草稿 | 正在撰写中 |
| 待检查 | 提交给 AI 检查 |
| 确认完善 | AI 检查后，用户确认无需修改 |
| 待开发 | 准备提交给开发 |
| 开发中 | 已交接给开发 |
| 已完成 | 开发完成 |

## 五、非功能需求

### 5.1 性能

- 页面加载 < 1 秒
- AI 检查响应 < 10 秒（取决于 AI API 速度）

### 5.2 数据

- 本地 MySQL 持久化存储
- 支持数据导出（JSON）

### 5.3 AI 模型

- 支持主流模型接入（OpenAI、Claude、国产模型）
- 支持自定义模型（用户自行提供 API）
- API Key 仅存服务端

## 九、附录

### 9.1 AI 检查提示词（示例）

```
你是一个需求文档审查助手。请检查以下需求文档，指出需要补充的地方。

## 检查维度

1. 完整性 - 必填项是否填写
2. 清晰度 - 描述是否说清楚
3. 逻辑闭环 - 流程是否完整
4. 边界情况 - 异常情况是否考虑
5. 分工明确 - 前后端各自要做什么

## 输出格式

请按以下 JSON 格式返回：
{
  "passed": true/false,
  "suggestions": [
    {"type": "completeness", "message": "建议补充..."},
    {"type": "clarity", "message": "建议说明..."}
  ]
}

## 需求内容
---
{需求描述}
---
```

### 9.2 待确认问题

2. 是否需要用户登录？（MVP 阶段可不做）
3. 数据备份策略？
